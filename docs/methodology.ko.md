# OOPforge 방법론 — 한국어 개념 가이드

이 문서는 OOPforge의 **개념과 프로세스**를 한국어로 설명한다.

> **정본은 영어다.** 실제 에이전트가 읽는 규칙은 `skills/`(영어)에 있다. 이 가이드는 스킬을
> 한 줄씩 번역한 미러가 아니라 **개념 설명**이다 — 미러는 드리프트하지만 개념 가이드는 안정적이다.
> 스킬 본문을 바꿔도 이 문서는 개념이 바뀔 때만 갱신한다.

---

## 1. OOPforge가 무엇인가

OOPforge는 **백엔드 OOP/DDD에 수직 특화된 방법론 + 아키텍처 강제(fitness function) 팩**이다.
범용 코딩 에이전트가 아니라, 그 위에 얹혀 **구조를 기본값으로 만드는 하네스**다.

- 대상 스택: **Java(Spring) / Python(FastAPI)** 백엔드.
- 푸는 문제: 제약이 없으면 AI 에이전트가 레이어/도메인 경계를 무너뜨린다(God Service, anemic domain).
  명시적 제약(스킬 + 하드룰 + CI 강제 + 휴먼 체크포인트)을 주면 위반이 거의 0으로 떨어진다.
- 핵심 지표는 토큰 절감이 아니라 **아키텍처 위반율**과 **재작업률**이다.

용어 대응: skills = 문법, 하드룰 = 린트, `examples/` = 표준 구현, install·commands = 런타임.

---

## 2. 워크플로 (코드보다 문서가 먼저)

새 도메인이나 큰 기능은 코드로 바로 점프하지 않는다. 각 단계 끝에 **사람 승인 체크포인트**가 있다.

```
Discovery → Design → Delivery Plan → Skeleton → Implement → Test
 (용어집)    (시그니처)   (범위/순서)     (빈 골격)    (한 유스케이스)  (검증)
```

| 단계 | 산출물 | 하지 말 것 |
|---|---|---|
| Discovery | 용어집, 바운디드 컨텍스트, 액터, 열린 질문 | 코드 |
| Design | 유스케이스 시그니처, 애그리거트 윤곽, 포트 | 구현 |
| Delivery Plan | 범위, 계약, 구현 순서, 테스트/릴리스 계획 | 코딩 |
| Skeleton | 패키지, 인터페이스, 빈 클래스 | 비즈니스 로직 |
| Implement | 한 번에 유스케이스 하나 | 여러 개 동시 |
| Test | 단위/통합/E2E | 테스트 없는 도메인 로직 |

작은 작업(단일 Value Object, 기존 도메인 확장, 리팩터, 리뷰)은 전체 파이프라인을 강제하지 않고
**Craft**가 가장 작은 경로를 고른다.

---

## 3. Craft — 단일 진입점

`/oopforge:craft <요청>`이 사용자 진입점이다. Craft는:

1. 요청을 분류해 **가장 작은 실행 경로**를 고른다.
2. 결정이 비어 있으면(언어·아키텍처·영속성·CQRS 여부 등) **모호성 해소**로 채운다 —
   심문하지 않고 기본값을 제시하고, 언어·아키텍처는 스택 게이트(Java-Spring/Python-FastAPI)를 거친다.
3. 비즈니스 로직 전에 **OOP Contract**(Use Case, Aggregate Root, 불변식, 상태 전이, 포트, 트랜잭션 경계)를 채운다.
4. 구현·테스트 후 **하드룰**을 검증하고 완료 보고를 남긴다.

advisory(추천만) 요청이면 구현하지 않고 가장 작은 경로만 추천한다.

## 4. 원칙 (oop-discipline)

백엔드 OOP 작업 전에 읽는 핵심 원칙. 영어 정본은 `skills/principles/oop-discipline.md`.

1. **행동은 도메인에 산다** — 규칙은 Service가 아니라 도메인 객체의 행동 메서드에.
2. **코드 전에 경계** — 애그리거트 루트·불변식·상태 전이·포트·트랜잭션 경계를 먼저 적는다.
3. **도메인은 프레임워크를 모른다** — 기술 세부는 어댑터 밖으로.
4. **CRUD보다 유스케이스** — 외부 메서드는 비즈니스 동사로.
5. **다른 애그리거트는 ID로 참조** — 객체 그래프를 경계 너머로 묶지 않는다.
6. **버그 수정 전에 실패 테스트** — 재현부터, 그다음 최소 수정.
7. **추상화 전에 빼라 — 작성 전 사다리** (아래 5절).
8. **교훈은 구조로 남긴다** — 같은 실수를 두 번 설명하면 테스트·린트·예제로.
9. **잘못된 추상화보다 중복** (아래 6절).
10. **코딩 전에 가정을 드러낸다** — 해석이 여러 개면 조용히 고르지 말고 Assumptions를 먼저 적는다(Craft Ambiguity resolution과 짝).
11. **외과수술식 수정만** — 요청에 필요한 곳만 건드린다. 인접 코드·주석·포맷 "개선" 금지. 내가 만든 orphan만 정리.

## 5. 작성 전 사다리 (YAGNI)

코드를 짜는 것은 마지막 수단이다. 무언가 만들기 전에 위 칸부터 밟고, 처음 해결되는 칸에서 멈춘다.

```
1. 존재할 필요가 있나?       → 아니면 안 만든다 (YAGNI)
2. 표준 라이브러리·언어 기능?  → 쓴다
3. 프레임워크 기본 제공?       → 쓴다 (Spring/FastAPI 내장)
4. 이미 깐 의존성?           → 쓴다
5. 한 줄·한 메서드로 되나?     → 거기서 끝낸다
6. 그래도 안 되면           → 동작하는 최소만 직접 짠다
```

- **본질 vs 우발**: 사다리는 우발적 복잡성(불필요한 추상화·중복·죽은 코드)만 깎는다.
  애그리거트 경계·불변식·포트 같은 **본질적 구조는 면제**.
- **게으르지만 태만하지 않다**: 신뢰 경계 검증·데이터 손실·보안은 어느 칸에서도 생략 금지.
- **미룬 것은 표식**: 의도적으로 최소만 했으면 upgrade path를 남긴다.

(ponytail의 YAGNI ladder를 백엔드 OOP/DDD 맥락으로 차용. MIT.)

## 6. DRY는 DDD 가드레일로

DRY는 **지식(규칙)의 중복**을 없애는 것이지, 닮아 보이는 코드를 무조건 합치는 게 아니다.
잘못된 추상화를 되돌리는 비용이 중복 비용보다 크다.

- **Rule of Three** — 두 번째 중복은 참고, 세 번째에야 추상화.
- **바운디드 컨텍스트를 가로질러 도메인 모델을 공유하지 않는다** — 닮아도 따로 진화하므로 중복이 옳다.
- 진짜 합칠 것은 흩어진 **비즈니스 규칙** — 도메인 행동 메서드 한 곳으로. (God Service·anemic domain 예방)

## 7. 하드룰 (측정 가능)

영어 정본은 `AGENTS.md`. 측정 가능한 한도로 둔 이유는 리뷰 부담과 에이전트 컨텍스트 크기 때문이다.

- 파일 **300줄 이하**, 메서드 **20줄 권장**, 스킬 **200줄 이하**.
- 도메인 레이어 프레임워크 import **0**.
- public setter 금지 — 팩토리 메서드나 의도를 드러내는 행동 메서드.
- 경계를 넘는 컬렉션은 방어 복사 또는 불변.
- 다른 애그리거트는 ID로만 참조.
- 테스트 없이 도메인 로직 커밋 금지.
- **외과수술식 수정** — 요청에 필요한 것만; 드라이브바이 리팩터·포맷·주석 손질 금지. 이번 변경이 만든 orphan만 정리.
- 레이어는 각각 별도 폴더(`controller/ service/ repository/ domain/`). Controller→Repository 직접 호출 금지.
- CI(`scripts/ci/archlint.py` + import-linter/ArchUnit)가 위반 시 PR을 막는다.
